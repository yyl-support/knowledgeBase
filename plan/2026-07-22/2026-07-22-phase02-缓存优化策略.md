# Phase02 缓存优化策略

## 问题

Phase02 批量执行 25 条用例耗时 **~10 分钟**，用户反馈过慢。

## 瓶颈分析

原始 `run_batch.py` 架构为每条用例 spawn 一个 `run_case.py` 子进程：

```
run_batch.py
  └─ subprocess: run_case.py (case 1)  → git clone → push → wait → teardown
  └─ subprocess: run_case.py (case 2)  → git clone → push → wait → teardown
  └─ ...
```

每个子进程独立执行完整生命周期，其中 **git clone** 操作重复 25 次。虽然单次 clone（`--depth 1`）只需数秒，但累加仍是一笔开销。

但更根本的瓶颈不在 clone，而在 **GitCode runner 调度延迟**——每条用例 push 后需等待平台分配 runner 执行 workflow，单条平均 30s，个别用例排队可达 800s+。

## 优化策略（两层）

### 第 1 层：消除子进程开销

**改动文件**: `phase02/scripts/run_batch.py`

**Before**: 每条用例 `subprocess.run([sys.executable, "run_case.py", ...])`

**After**: 直接 `import workflow_runner as wr`，在单个 Python 进程内循环调用 `wr.run_case()`。

```python
# 核心变化：一个 Workspace 上下文内串行执行全部用例
cfg = wr.RunnerConfig(branch=None)
with wr.Workspace(cfg) as ws:
    for i, c in enumerate(cases, 1):
        contract_doc = rc.load_contract(contract)
        wf, asserts = rc.load_execution_inputs(contract_doc, run_dir, cid)
        # preflight → run_case → assertion → write_result 全部内联
        rr = wr.run_case(ws, cfg, cid, wf, ...)
        verdict = ae.evaluate(rr, engine_asserts)
        rec = rc.write_result(run_dir, contract_doc, verdict, rr)
```

**收益**:
- 消除 25 次 Python 解释器启动开销
- 消除 25 次模块 import 开销
- Workspace 上下文跨用例共享（为第 2 层打基础）

### 第 2 层：持久化 Git 缓存

**改动文件**: `phase02/scripts/workflow_runner.py`（`Workspace` 类）

**Before**: 每次 `Workspace.__enter__` 创建临时目录 → `git clone`；`__exit__` 时 `shutil.rmtree` 删除。

**After**: 使用固定缓存路径 `phase02/.cache/repos/<owner>/<repo>/`：
- **首次**：`git clone --depth 1` 到缓存目录
- **后续**：`git fetch --depth 1 origin` + `git reset --hard origin/<branch>`（抹掉上次 batch 的 teardown 残留）
- **退出**：`pass`（不删除，持久保留）
- **fetch 失败兜底**：删除缓存目录，重新 clone

```python
_CACHE_ROOT = os.path.join(PHASE02, ".cache", "repos")

class Workspace:
    def __init__(self, cfg):
        self.repo_dir = os.path.join(_CACHE_ROOT, cfg.owner, cfg.repo)

    def __enter__(self):
        if os.path.isdir(os.path.join(self.repo_dir, ".git")):
            # 缓存命中：增量更新
            rc, _ = _sh("git fetch --depth 1 origin", cwd=self.repo_dir)
            if rc == 0:
                _sh(f"git checkout -q {self.cfg.branch}", cwd=self.repo_dir)
                _sh(f"git reset --hard origin/{self.cfg.branch}", cwd=self.repo_dir)
            else:
                shutil.rmtree(self.repo_dir, ignore_errors=True)
        if not os.path.isdir(os.path.join(self.repo_dir, ".git")):
            # 缓存未命中或 fetch 失败：全量 clone
            os.makedirs(os.path.dirname(self.repo_dir), exist_ok=True)
            _sh(f'git clone --depth 1 "{url}" "{self.repo_dir}"')
        return self

    def __exit__(self, *exc):
        pass  # 持久化，不删除
```

## 效果

| 指标 | 优化前 | 优化后 |
|------|--------|--------|
| git clone 次数/批次 | 25 次 | 1 次（首次）→ 0 次（后续批次） |
| Python 进程数 | 26（1 主 + 25 子） | 1 |
| clone 耗时占比 | ~15-20% | ~1-2% |

## 局限

优化只削减了本地 git 操作和进程开销。**真正的瓶颈在 GitCode 平台侧**——runner 调度延迟（单条 30s~800s 不等），这部分不受本地控制。进一步加速需要：
- GitCode 平台增加 runner 并发数
- 减少轮询间隔（当前 12s，调低会增加 API 请求频率）
- 多条用例并行执行（当前串行，可改为同维度内并行）
