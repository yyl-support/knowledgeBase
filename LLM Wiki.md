# LLM Wiki

> 原文：Andrej Karpathy，[llm-wiki](https://gist.github.com/karpathy/442a6bf555914893e9891c11519de94f)

一种利用 LLM 构建个人知识库的方法。这是一份想法文档，可以直接复制给 OpenAI Codex、Claude Code、OpenCode / Pi 等 LLM Agent；具体实现细节由 Agent 与使用者协作确定。

## 核心理念

大多数人使用 LLM 和文档的方式类似 RAG（检索增强生成）：上传一批文件，在提问时由 LLM 检索相关片段并生成答案。这种方式有效，但每次提问时，LLM 都要从头重新发现知识，没有积累。

面对需要综合五份文档的细微问题，LLM 每次都必须重新找到并拼接相关片段，成果并未沉淀。NotebookLM、ChatGPT 文件上传功能以及大多数 RAG 系统都属于这种模式。

这里的想法不同：不只在查询时从原始文档中检索，而是让 LLM **持续、增量地构建和维护一个持久化 Wiki**。该 Wiki 是位于使用者与原始资料之间的一组结构化、相互链接的 Markdown 文件。

当加入一份新资料时，LLM 不只是为未来检索建立索引，而是会阅读、提取关键信息，并整合到已有 Wiki：更新实体页面、修订主题摘要、记录新数据与旧观点间的矛盾，并强化或质疑正在形成的综合结论。知识只需被整理一次，之后持续保持更新，而不是每次提问时重新推导。

关键区别是：**Wiki 是一个持久的、会复利增长的产物。** 交叉引用已经建立，矛盾已经标记，综合结论已经反映读过的资料。每增加一个来源、每提出一个问题，Wiki 都会更加丰富。

使用者无需，或很少需要，亲自撰写 Wiki。LLM 负责摘要、交叉引用、归档和记录维护；使用者负责筛选资料、探索和提出正确的问题。实践中，可以一侧打开 LLM Agent，另一侧打开 Obsidian：LLM 根据对话编辑文件，使用者实时浏览页面、跟随链接、查看关系图。Obsidian 是 IDE，LLM 是程序员，Wiki 则是代码库。

适用场景包括：

- **个人**：追踪目标、健康、心理状态与自我提升；归档日记、文章和播客笔记，逐步形成结构化的自我图景。
- **研究**：围绕某主题持续数周或数月，阅读论文、文章和报告，逐步建立带有演化论点的综合 Wiki。
- **阅读书籍**：随章节归档，为人物、主题、情节线及其关系建立页面；读完后获得一个丰富的辅助 Wiki。
- **企业或团队**：由 LLM 维护内部 Wiki，输入来自 Slack 讨论、会议记录、项目文档和客户沟通，人工可参与审核。
- **竞品分析、尽职调查、旅行规划、课程笔记与爱好研究**：任何长期积累知识，并希望资料被系统整理而非散落的场景。

## 架构

系统分为三层：

**原始来源（Raw sources）**：精心收集的源文档，包括文章、论文、图片和数据文件。这些资料不可变，LLM 只读取而不修改，是事实来源。

**Wiki**：由 LLM 生成的 Markdown 文件目录，包括摘要、实体页面、概念页面、比较、概览与综合分析。LLM 创建页面、在新资料到来时更新页面、维护交叉引用并保证整体一致；使用者阅读，LLM 写作。

**模式定义（Schema）**：一个文档，例如 Claude Code 的 `CLAUDE.md` 或 Codex 的 `AGENTS.md`。它规定 Wiki 结构、约定，以及导入资料、回答问题、维护 Wiki 时的工作流。这是关键配置文件，使 LLM 成为有纪律的 Wiki 维护者，而非普通聊天机器人。使用者和 LLM 可以随经验共同演化它。

## 操作

### 导入（Ingest）

将新资料放入原始资料库并要求 LLM 处理。典型流程为：LLM 阅读资料，与使用者讨论关键收获，在 Wiki 中撰写摘要页，更新索引，更新相关实体页和概念页，并在日志中追加记录。一份来源可能会影响 10 到 15 个 Wiki 页面。

可以一次导入一份资料并保持参与，阅读摘要、检查更新、指导 LLM 强调重点；也可以批量导入大量资料并减少监督。适合自己的流程应写入模式定义，供之后会话使用。

### 查询（Query）

向 Wiki 提问时，LLM 搜索相关页面、阅读它们，并生成带引用的综合回答。回答可按问题需要形成 Markdown 页面、对比表、Marp 幻灯片、matplotlib 图表或 Canvas 画布。

重要洞见是：**高质量回答可以归档回 Wiki，成为新页面。** 所请求的对比、分析和新发现的关联都很有价值，不应消失在聊天记录中。这样探索本身也会像导入资料一样持续沉淀到知识库。

### 检查（Lint）

定期让 LLM 对 Wiki 做健康检查，关注：

- 页面之间的矛盾。
- 已被较新来源推翻的陈旧观点。
- 没有入链的孤立页面。
- 被提及却没有独立页面的重要概念。
- 缺失的交叉引用。
- 可通过网络搜索填补的数据缺口。

LLM 也擅长建议下一步值得研究的问题和可寻找的新资料，使 Wiki 在增长过程中保持健康。

## 索引与日志

两个特殊文件帮助使用者和 LLM 在 Wiki 增长时导航。

### `index.md`

`index.md` 面向内容，是 Wiki 全部内容的目录。每个页面列出链接、一行摘要以及可选元数据，例如日期或来源数量，并按实体、概念、来源等类别组织。

LLM 每次导入时更新索引。回答问题时，先读取索引定位相关页面，再深入阅读对应内容。这种方式在约 100 个来源、数百个页面的中等规模下已足够有效，也避免了搭建嵌入式 RAG 基础设施的需求。

### `log.md`

`log.md` 面向时间，是发生事项及时间的仅追加记录，包括资料导入、查询和检查。每条记录宜使用一致前缀，例如：

```md
## [2026-04-02] ingest | 文章标题
```

这样可通过简单 Unix 工具获取最近记录：

```bash
grep "^## \[" log.md | tail -5
```

日志提供 Wiki 演化的时间线，并帮助 LLM 了解最近完成的工作。

## 可选：CLI 工具

当 Wiki 发展到一定阶段，可构建小工具，让 LLM 更高效地操作 Wiki。最直接的是页面搜索引擎：规模较小时索引文件已足够，规模增长后则需要更完善的搜索。

[qmd](https://github.com/tobi/qmd) 是一个选择。它是 Markdown 文件的本地搜索引擎，提供混合 BM25 / 向量搜索与 LLM 重排序，均可在设备本地运行。它既提供供 LLM 在 Shell 中调用的 CLI，也提供作为原生工具使用的 MCP Server。也可以按需要自行构建简单搜索脚本。

## 技巧与工具

- **Obsidian Web Clipper**：将网页文章转换为 Markdown 的浏览器扩展，便于快速将资料放入原始资料库。
- **下载图片到本地**：在 Obsidian 的“设置 -> 文件与链接”中，将附件文件夹设为固定目录，例如 `raw/assets/`；随后在“设置 -> 快捷键”中找到“下载当前文件的附件”并绑定快捷键，例如 `Ctrl+Shift+D`。剪藏文章后即可下载其图片。
- 本地图片是可选的，但能让 LLM 直接查看和引用图片，不必依赖可能失效的 URL。LLM 无法在一次读取中原生理解带内嵌图片的 Markdown；可先读文本，再分别查看部分或全部引用图片以获取补充上下文。
- **Obsidian 的关系图谱视图**：用于观察 Wiki 整体结构、页面间关联、枢纽页面和孤立页面。
- **Marp**：基于 Markdown 的幻灯片格式，Obsidian 有对应插件，可直接从 Wiki 内容生成演示文稿。
- **Dataview**：可查询页面 Frontmatter 的 Obsidian 插件。若 LLM 为页面添加 YAML Frontmatter，例如标签、日期和来源数量，Dataview 可生成动态表格和列表。
- Wiki 本质上是一个由 Markdown 文件组成的 Git 仓库，因此可免费获得版本历史、分支管理和协作能力。

## 为什么有效

维护知识库最枯燥的部分不是阅读或思考，而是记录维护：更新交叉引用、保持摘要最新、记录新数据何时与旧观点矛盾，以及在数十个页面之间维持一致性。

人们放弃 Wiki，是因为维护负担增长得比价值更快。LLM 不会厌倦，不会忘记更新交叉引用，并且一次可以修改 15 个文件。因此，Wiki 可以得到持续维护，因为维护成本接近于零。

人的职责是筛选资料、引导分析、提出好问题，并思考信息意味着什么；LLM 的职责则是其余的一切。

这个理念在精神上与范内瓦·布什（Vannevar Bush）于 1945 年提出的 Memex 有关：一种私人维护的知识存储系统，文档之间通过关联路径连接。布什的愿景与这种模式比后来发展的 Web 更接近：私密、主动维护，并且文档之间的关联与文档本身同样有价值。他当时无法解决的问题是“谁来维护”，而 LLM 正好承担了这一工作。

## 说明

本文刻意保持抽象，描述的是一种模式，而不是具体实现。目录结构、模式约定、页面格式和工具链，都取决于领域、个人偏好和使用的 LLM。上述内容全部可选且模块化：选择有用的部分，忽略其他部分。

例如，纯文本资料不需要图片处理；小型 Wiki 只需索引文件而不必使用搜索引擎；如果不需要幻灯片，只保留 Markdown 页面即可。正确的使用方式是将这份思路提供给 LLM Agent，并与其协作形成适合自己需求的版本。本文唯一的职责是传达该模式，其余细节可由 LLM 与使用者共同解决。
