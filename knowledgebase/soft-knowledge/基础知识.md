# 基础知识

> Kubernetes 与集群相关的基础概念。和具体业务解耦，作为通用参考。

---

## 一、核心概念

### Pod 与 Node

| 概念 | 定位 | 类比 |
|------|------|------|
| **Node** | 物理机 / 虚拟机，提供 CPU、内存、GPU | 房子 |
| **Pod** | K8s 最小调度单位，一个或多个容器的壳 | 住在里面的人 |

- 一个 Pod **只能跑在一个 Node 上**，不能跨 Node
- 一个 Node 可以跑**多个 Pod**
- 调度器（scheduler）负责把 Pod 分配到合适的 Node
- 一旦绑定，Pod 就一直跑在那台 Node 上，直到结束或被驱逐

### 集群拓扑

```
集群 (Cluster)
├── 控制面 (Control Plane)
│   ├── kube-apiserver    ← 一切操作的入口，REST API
│   ├── etcd              ← 分布式 KV 存储，保存集群所有状态
│   ├── kube-scheduler    ← 为新建 Pod 选择 Node
│   ├── kube-controller-manager  ← 运行各种控制器（Deployment、Node 等）
│   └── cloud-controller-manager ← 云平台对接（可选）
│
├── Node-A
│   ├── kubelet           ← Node 管家，与控制面通信，管理容器生命周期
│   ├── kube-proxy        ← 网络代理，维护 Node 上的网络规则
│   └── 容器运行时         ← containerd / CRI-O，真正跑容器的东西
│       ├── Pod-1
│       └── Pod-2
│
└── Node-B
    └── ...
```

### Namespace

虚拟集群隔离。同一集群内，不同 Namespace 下的资源互不可见（默认）。

```bash
kubectl get pods -n <namespace>
```

常见系统 Namespace：`kube-system`（核心组件）、`kube-public`（公开信息）、`default`（默认）

## 二、工作负载资源

### Pod

最小调度单元。**不直接创建 Pod**，而是通过高级控制器管理。

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: nginx
spec:
  containers:
  - name: nginx
    image: nginx:latest
    ports:
    - containerPort: 80
```

### Deployment

管理**无状态应用**。声明期望副本数，自动创建/替换 Pod。支持滚动更新和回滚。

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: my-app
spec:
  replicas: 3          # 同时跑 3 个副本
  selector:
    matchLabels:
      app: my-app
  template:            # Pod 模板
    metadata:
      labels:
        app: my-app
    spec:
      containers:
      - name: app
        image: my-app:v1
```

### StatefulSet

管理**有状态应用**（数据库、消息队列等）。每个 Pod 有稳定标识（`pod-0`, `pod-1`...）和持久存储。

### DaemonSet

每个 Node 上跑**恰好一个** Pod。用于日志采集、监控 agent、网络插件等。

### Job / CronJob

Job：跑一次就结束的任务。
CronJob：按 cron 表达式周期性创建 Job。

## 三、网络

### Service

为一组 Pod 提供**稳定的访问入口**（Pod IP 会变，Service IP 不变）。

| 类型 | 用途 |
|------|------|
| ClusterIP | 集群内部访问（默认） |
| NodePort | 通过 Node IP + 端口对外暴露 |
| LoadBalancer | 云厂商 LB 对外暴露 |
| ExternalName | DNS 别名指向外部地址 |

```yaml
apiVersion: v1
kind: Service
metadata:
  name: my-svc
spec:
  selector:
    app: my-app       # 选择带此 label 的 Pod
  ports:
  - port: 80
    targetPort: 8080
```

### Ingress

七层 HTTP/HTTPS 路由。按域名/路径转发到不同 Service。

### 网络模型要点

- 所有 Pod 共享一个集群平面网络（Pod 之间可以直接 IP 通信，不跨 Node 也通）
- 同 Pod 内容器共享网络命名空间（`localhost` 互通）
- 每个 Pod 有独立 IP（不是每个容器有独立 IP）

## 四、存储

### Volume 类型

| 类型 | 生命周期 | 用途 |
|------|----------|------|
| emptyDir | 随 Pod | 临时数据，Pod 删了就没了 |
| hostPath | 随 Node | 挂载 Node 上的目录（生产慎用） |
| ConfigMap / Secret | 随 Pod | 配置注入 |
| PVC (PersistentVolumeClaim) | 独立于 Pod | 持久化存储 |

### PV / PVC 模型

- **PV (PersistentVolume)**：管理员预置的存储（或通过 StorageClass 动态创建）
- **PVC (PersistentVolumeClaim)**：用户申请存储的请求
- **StorageClass**：定义存储类型（SSD / HDD / NFS 等）

```
用户创建 PVC → K8s 找匹配的 PV 绑定 → Pod 引用 PVC 挂载
```

## 五、调度

### 调度流程

```
新建 Pod → 过滤（剔除不满足条件的 Node）
        → 打分（给剩余 Node 排名）
        → 绑定（选最高分 Node）
```

### 亲和性 / 反亲和性

| 类型 | 含义 |
|------|------|
| nodeAffinity | Pod 偏好某些 Node（如 SSD 节点） |
| podAffinity | Pod 偏好和某些 Pod 待在一起 |
| podAntiAffinity | Pod 偏好和某些 Pod **分开**（高可用） |

### 污点与容忍

- **Taint（污点）**：标记在 Node 上，驱逐不匹配的 Pod
- **Toleration（容忍）**：标记在 Pod 上，表示可以容忍这个污点

```
# 给 Node 打污点：只有带容忍的 Pod 能调度上来
kubectl taint nodes node1 gpu=true:NoSchedule
```

### 资源请求与限制

```yaml
resources:
  requests:      # 调度依据，保证至少有这么多资源
    cpu: "500m"
    memory: "512Mi"
  limits:        # 硬上限，超了会被 kill (OOM)
    cpu: "1"
    memory: "1Gi"
```

## 六、健康检查（Probe）

| 探针 | 用途 |
|------|------|
| livenessProbe | 容器是否活着？**死了就重启** |
| readinessProbe | 容器是否就绪？**未就绪就不接流量** |
| startupProbe | 容器是否启动完成？**启动中禁用前两种探针**（防止慢启动被误杀） |

```yaml
livenessProbe:
  httpGet:
    path: /healthz
    port: 8080
  initialDelaySeconds: 30   # 等 30 秒再开始探
  periodSeconds: 10         # 每 10 秒探一次
```

**关键时序陷阱**：如果 `initialDelaySeconds` 太短，Pod 还没启动完就被探针判定失败→重启→永远起不来（CrashLoopBackOff）。

## 七、RBAC（权限）

```
ServiceAccount（身份）
    └── Role / ClusterRole（能做什么）
        └── RoleBinding / ClusterRoleBinding（绑定到一起）
```

- **Role**：Namespace 级别权限
- **ClusterRole**：集群级别权限

```yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  namespace: default
  name: pod-reader
rules:
- apiGroups: [""]
  resources: ["pods"]
  verbs: ["get", "list"]
```

## 八、GitOps（ArgoCD）

核心理念：**Git 仓库是唯一真相来源**。集群状态 = Git 仓库里的 YAML。任何偏差都会被自动修正。

```
开发者提交 YAML 到 Git
    ↓
ArgoCD 检测到 Git 变更
    ↓
对比 Git 状态 vs 集群实际状态
    ↓
自动同步（有差异就用 Git 的覆盖）
```

核心组件：
- **argocd-server**：Web UI + API
- **argocd-repo-server**：连接 Git 仓库
- **argocd-application-controller**：持续监控和同步

```yaml
apiVersion: argoproj.io/v1alpha1
kind: Application
metadata:
  name: my-app
  namespace: argocd
spec:
  project: default
  source:
    repoURL: https://github.com/org/repo
    path: k8s/overlays/prod
  destination:
    server: https://kubernetes.default.svc
    namespace: prod
  syncPolicy:
    automated:
      prune: true        # 删除 Git 里不存在的资源
      selfHeal: true     # 集群里手动改的自动纠正
```

## 九、常用命令速查

```bash
# Pod
kubectl get pods -n <ns>       # 列出 Pod
kubectl describe pod <name>    # 看详情（事件在最后）
kubectl logs <pod> [-c <container>]  # 看日志

# 进入容器
kubectl exec -it <pod> -- /bin/sh

# 排查
kubectl get events -n <ns> --sort-by=.lastTimestamp  # 看最近事件
kubectl top pods -n <ns>       # 看资源用量

# 调试性 Pod（不经过 scheduler）
kubectl run debug --image=busybox --rm -it --restart=Never -- /bin/sh

# Node
kubectl get nodes               # 看所有 Node 状态
kubectl describe node <name>    # 看 Node 资源容量和已分配

# 配置
kubectl create configmap my-cm --from-file=config.yaml

# rollout
kubectl rollout status deployment/<name>
kubectl rollout undo deployment/<name>
```

## 十、扩展概念速查

| 概念 | 一句话说明 |
|------|-----------|
| CRD (CustomResourceDefinition) | 自定义 K8s 资源类型，扩展 API |
| Operator | CRD + 自定义 Controller，把运维知识写成代码自动执行 |
| Helm | K8s 的包管理器，把一组 YAML 打包成 Chart |
| Kustomize | 基于 YAML overlay 的 patch 式配置管理 |
| HPA (HorizontalPodAutoscaler) | 根据 CPU/内存指标自动扩缩 Pod 副本数 |
| VPA (VerticalPodAutoscaler) | 自动调整 Pod 的资源 requests/limits |
| Service Mesh | 在应用层之外统一管理服务间通信（如 Istio） |
| CNI | 容器网络接口，负责 Pod 网络（Calico / Flannel / Cilium） |
| CSI | 容器存储接口，负责持久化存储驱动 |
| MutatingWebhook | 在 Pod 创建前注入字段（如 sidecar、环境变量） |
