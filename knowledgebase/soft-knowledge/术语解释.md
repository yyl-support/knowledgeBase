# backlog .ai-flow 术语解释

> 创建时间：2026-06-10
> 用途：持续更新，记录 backlog .ai-flow 体系中所有术语的含义
> 约定：`(+)` 标记表示该术语来自 backlog 项目内部定义，含义可能与通用定义有出入

---

## 1. Workflow（GitHub Actions 工作流）

### 通用定义

Workflow 是 GitHub Actions 的**自动化流水线单元**。一个 Workflow 就是一个 YAML 文件（放在 `.github/workflows/` 目录下），定义了"当某个事件发生时，执行哪些步骤"。

### 在 backlog 中的具体含义 (+)

backlog 中的 Workflow 分两类：

| 类型 | 示例文件 | 触发方式 | 职责 |
|------|---------|---------|------|
| **Router Workflow** | `command-router.yml` | `issue_comment.created` 事件（每条评论都触发） | 唯一入口，统一监听 Issue 评论，正则匹配后分发到下游 |
| **下游 Workflow** | `issue-1-analyze-requirement.yml`、`issue-2-implement.yml`、`issue-3-release.yml` | `workflow_dispatch`（仅被 router 调起） | 执行具体的 AI Agent 编排任务 |

### 为什么需要 Router？

传统做法是每个下游 Workflow 都直接监听 `issue_comment.created` 事件，这会导致**每条评论触发所有 Workflow 都跑一遍**——尽管 99% 的评论（如讨论、反馈）不匹配任何命令，但每个 Workflow 都会产生一条 `skipped` 运行记录，造成大量噪音。

backlog 通过**单一入口 Router** 解决了这个问题：
- Router 是唯一监听 Issue 事件的 Workflow
- 下游 Workflow 全部改为 `workflow_dispatch` 触发（不直接监听事件）
- Router 用 PAT（`GH_TOKEN`）调起 `gh workflow run` 来触发下游

### 关键约束

- Router 必须使用 **PAT**（Personal Access Token）而非默认的 `GITHUB_TOKEN`，因为 `GITHUB_TOKEN` 触发的 `workflow_dispatch` 不会真正启动新的 workflow run
- 回评到 Issue 则使用 `GITHUB_TOKEN`（权限分离）

### 关系图

```
GitHub 推送 issue_comment.created 事件
        │
        ▼
command-router.yml（唯一监听者）
        │
        ├─ /ai-requirement-analysis → gh workflow run issue-1-analyze-requirement.yml
        ├─ /ai-develop-preview      → gh workflow run issue-2-implement.yml (PHASE=preview)
        ├─ /ai-develop-submit       → gh workflow run issue-2-implement.yml (PHASE=submit)
        ├─ /ai-deploy-test          → gh workflow run issue-3-release.yml
        └─ 不匹配任何命令           → 忽略，不触发任何下游
```

## 2. 标签驱动的流程控制

### 概述

backlog 的整个 AI 开发流程是**由 GitHub Issue 的标签（Label）驱动的**。标签不仅是分类标记，更是流程的**状态机触发器**。

### 标签的三层作用

#### 第一层：归属路由 — `project:<umbrella>`

```yaml
# 在 .ai-flow/services/<id>.yaml 中定义
service:
  label: project:forum-reply-robot   # ← 打这个标签就路由至此服务
```

- 当 Issue 被打上 `project:forum-reply-robot` 标签时，`resolve_service.py` 会匹配到对应服务的 YAML 配置
- 输出该服务的仓库地址、集群、命名空间等部署参数
- **这是整个流程的起点**：没有这个标签，AI Agent 不知道代码在哪

#### 第二层：阶段流转 — `accepted` / `need_design` / `need_security` 等

| 标签 | 触发行为 |
|------|---------|
| `accepted` | 自动触发 `/ai-develop-preview`（开发预览阶段） |
| `need_design` | 强制要求产出设计文档，否则不能进入 dev 阶段 |
| `need_security` | 门禁阶段强制跑全量 security-gate（而非仅 gitleaks） |
| `need_itest` | 要求跑集成测试 |
| `project:<xxx>` | 路由到具体服务配置 |

流程状态机（简化）：

```
Issue 创建
  │
  ├─ 自动打 project:xxx 标签（AI 分析归属）
  │
  ├─ 打 accepted → 触发开发预览
  │
  ├─ [缺陷]/[任务] 标题 → 跳过需求分析，直接开发
  │
  └─ [需求] 标题 → 走全流程（需求分析→设计→开发→测试→发布）
```

#### 第三层：流程类型分流 — Issue 标题语义

| 标题前缀 | 含义 | 流程差异 |
|---------|------|---------|
| `[需求]` | 新功能需求 | 走完整五阶段（需求分析→开发预览→开发提交→测试发布→上线） |
| `[缺陷]` | Bug 修复 | 跳过需求分析阶段，不产出设计文档 PR |
| `[任务]` | 工程任务 | 同缺陷，跳过分析阶段 |

### 为什么用标签驱动而非 Webhook？

标签驱动的好处：
- **可见性强**：任何人打开 Issue 就能看到当前流程状态
- **人工可控**：管理员可以手动加/删标签来干预流程
- **解耦**：标签变化是标准的 GitHub 事件，不需要额外的回调系统
- **弹性**：标签可以叠加（如同时有 `need_design` + `need_security`），控制更细粒度

## 3. 冒烟测试 vs 功能测试 vs 其他测试类型

### 3.1 冒烟测试（Smoke Testing）

**定义**：在部署后立即执行的一组**最小化验证**，只检查"系统能不能跑起来"。

**在 backlog 中的具体含义** (+)：
- `tester` Agent 在 K8s 预览 Pod 部署完成后，对预览环境的关键端点发 1-2 个请求
- 只验证核心路径是否通（如 `/health` 返回 200、主功能接口不报 500）
- **1 轮 best-effort**：过就过了，不过也不重试
- 目的：**快速阻断明显故障**，不让坏的预览部署留在那浪费资源

### 3.2 功能测试（Functional Testing）

**定义**：验证系统的**业务逻辑是否符合需求规格**。不关心内部实现，只关心输入→输出是否正确。

**与冒烟测试的区别**：

| 维度 | 冒烟测试 | 功能测试 |
|------|---------|---------|
| 目标 | 验证系统能跑 | 验证功能正确 |
| 覆盖范围 | 1-2 个核心端点 | 完整的功能矩阵 |
| 深度 | 浅（只测"通不通"） | 深（测"对不对"） |
| 用例数 | 3-5 条 | 几十到几百条 |
| 失败后的行为 | 标记部署失败，不回滚 | 标记功能缺陷，需要修复 |
| 执行时机 | **每次部署后立即跑** | 部署稳定后全面跑 |
| backlog 中对应 | tester Agent 的冒烟模式 | tester Agent 的完整测试模式 + integration-tests |

### 3.3 CI/CD 中的测试金字塔

```
          ╱  E2E 测试 ╲          ← 最慢、最贵、最接近用户
         ╱  集成测试    ╲
        ╱   功能测试     ╲
       ╱    组件测试      ╲
      ╱     单元测试       ╲    ← 最快、最便宜、数量最多
     ╱      冒烟测试        ╲
```

backlog 中各类测试的对应关系：

| 测试类型 | 英文 | 在 backlog 中的对应 | 执行阶段 | 执行者 |
|---------|------|-------------------|---------|--------|
| **冒烟测试** | Smoke Test | `tester` Agent 冒烟模式 | Phase B1 preview 部署后 | tester Agent |
| **单元测试** | Unit Test | `go test` / `pytest` / `vitest` | Phase B2 submit 门禁阶段 | gates/ + tester Agent |
| **组件测试** | Component Test | — | 暂未独立 | — |
| **功能测试** | Functional Test | `tester` Agent 完整测试模式 | Phase B2 submit 阶段 | tester Agent |
| **集成测试** | Integration Test | `integration-tests/services/` | Phase B2 submit 后、Phase C 部署后 | `run_all.sh` |
| **端到端测试** | E2E Test | 业务仓库内的 E2E 用例 | Phase C 发布后 | — |
| **安全扫描** | Security Scan | `security-gate`（Gitleaks + Trivy + SAST） | Phase B2 开 PR 前 | gates/ |
| **回归测试** | Regression Test | `run_regression_eval.py`（RAG 质量趋势追踪） | 每周定时 或 重大变更后 | GitHub Actions CronJob |

### 3.4 各测试在 CI/CD 五个阶段的分布

```
Phase A: 需求分析        → 无测试
Phase B1: 开发预览       → 冒烟测试（1轮 best-effort）
Phase B2: 开发提交       → 单元测试 + 安全扫描 + 功能测试 + 集成测试
Phase C: 测试发布        → 集成测试 + E2E 测试
Phase D (不存在的阶段)    — 
Phase E (正式上线后)     → 回归测试（定期） + 监控告警
```

## 4. Best-Effort（尽力而为模式）

### 通用定义

Best-Effort 是一种**不保证结果、但尽力尝试**的执行策略。在计算机领域，它指"我尽量做，但如果失败了也不会重试或报错阻塞"。

### 在 backlog 中的具体含义 (+)

backlog 在两处使用了 best-effort 语义：

#### 4.1 冒烟测试的 best-effort

```
preview 段的 tester（冒烟测试）：
  - 只跑 1 轮
  - PASS → 正常流程继续
  - FAIL → 也继续！不阻塞后续流程
  - FINAL_STATUS = "preview_partial"（部分成功）
```

**设计意图**：预览阶段的目的是"让人快速看到一个能跑的东西"。如果冒烟失败，说明部署有问题，但不值得为此重试——因为预览本质上是给**人工看的草稿**，不是交付物。

#### 4.2 Submit 段的默认 best-effort（无对抗模式）

```
submit 段（无 --allgate）：
  - 只跑 1 轮
  - 门禁只跑 gitleaks（敏感信息检查），其余 5 项跳过
  - FAIL → 也开 PR！回评告知哪个检查挂了
  - 不进入 repair 循环
```

**设计意图**：保持流水线快速（不因某项非关键检查失败而反复重试），把完整检查留给 PR 阶段的 CI。这是一种**不能因为无法完美而阻塞交付**的务实哲学。

### 对比：Best-Effort vs 严格模式

| 维度 | Best-Effort | 严格模式 |
|------|------------|---------|
| 测试轮数 | 1 轮 | 最多 5 轮 |
| 失败后 | 继续/开 PR | 修复后重试 |
| 门禁 | 仅 gitleaks | 全部 7 项 |
| 使用场景 | preview、默认 submit | `ADVERSARIAL_MODE=1` 或 `--allgate` |
| 设计意图 | 快速迭代、不阻塞 | 严格质量保障 |

## 5. 对抗模式（Adversarial Mode）

### 通用定义

"对抗"（Adversarial）在 AI/Machine Learning 领域指**两个系统互相博弈**以提升各自能力。在软件工程中，它指**攻击性测试**——故意找代码的漏洞和边界 case。

### 在 backlog 中的具体含义 (+)

backlog 的对抗模式是一个**多轮 review-repair 循环**：

```
对抗模式开启条件：ADVERSARIAL_MODE=1 环境变量
```

```
对抗模式流程（最多 5 轮）：

Round 1:
  design → dev → deploy → gates → review → tester
     │                                              │
     │  FAIL                                       PASS → 开 PR
     ▼
  写 feedback.md（reviewer Agent 的评审意见）
     │
     ▼
Round 2:
  design Agent 读 feedback.md → 修改设计
  dev Agent 按新设计改代码
  → deploy → gates → review → tester
     │
     │  FAIL → 再写 feedback.md
     ▼
Round 3, 4, 5...
     │
     ▼
  用尽 MAX_FIX_ROUNDS → 开 PR（附带所有未修复问题的说明）
```

### 对抗模式的本质

这不是真正的"对抗攻击"，而是一个 **reviewer Agent 扮演"挑剔的审查者"角色，design/dev Agent 扮演"修复者"角色** 的多轮博弈：

| 角色 | 行为 | 类比 |
|------|------|------|
| **reviewer Agent** | 发现代码问题，写 feedback.md | 代码审查中的审查者 |
| **design Agent** | 读 feedback，修改设计方案 | 架构师响应审查意见 |
| **dev Agent** | 按新设计改代码 | 开发者修复审查意见 |

### 为什么叫"对抗"？

这个命名来自 AI 领域的 GAN（生成对抗网络）概念——但实际更接近 **"对抗性代码审查"**（Adversarial Code Review），即用挑剔视角深挖问题。名字听起来比较酷，实质是一个**带反馈回路的迭代改进机制**。

### 对抗模式 vs 默认模式

| 维度 | 默认模式（Best-Effort） | 对抗模式 |
|------|----------------------|---------|
| 修复轮次 | 1 轮 | 最多 5 轮 |
| feedback.md | 不生成 | 每轮 FAIL 后生成 |
| 门禁 | 仅 gitleaks | 可配置（默认同，可用 `--allgate` 全开） |
| reviewer 作用 | 单次评审 | 多轮迭代，每轮输出改进建议 |
| 适用场景 | 简单改动、快速预览 | 复杂需求、高可靠性要求 |
| 设计文档 | 不改 | 每轮可能修改（基于 reviewer 反馈） |

### 配置方式

```bash
# 开启对抗模式
ADVERSARIAL_MODE=1

# 对抗模式下也跑全部门禁
ADVERSARIAL_MODE=1
# 配合 /ai-develop-submit --allgate
```

## 6. Gate（门禁）与 Security Gate（安全门禁）

### 通用定义

在 CI/CD 领域，**Gate** 是指流水线中的**强制检查点**——代码必须通过该检查点才能继续往下走。如果失败，流水线被阻断（fail-fast），不允许进入下一阶段。

**Security Gate** 特指安全相关的检查集合：密钥扫描、漏洞扫描、许可证合规等。

### 在 backlog 中的具体含义 (+)

backlog 的 Gate 体系分两个层面：

#### 6.1 常规门禁（gates/）

由 `gates/run.sh` 驱动，共 **7 项检查**（`checks.sh` 实现）：

| # | 检查项 | 工具 | 说明 |
|---|--------|------|------|
| 1 | 敏感信息 | Gitleaks | 扫描硬编码密钥/token/密码 |
| 2 | 设计文档 | 文件存在性检查 | 检查 `issue_docs/` 下是否有设计文档 |
| 3 | 漏洞扫描 | `npm audit` / `pip audit` / `go list` | 依赖包的已知漏洞 |
| 4 | 安全编码 | SAST 静态分析 | 代码中的安全缺陷 |
| 5 | License | 许可证合规检查 | 依赖许可证是否与项目兼容 |
| 6 | 镜像漏洞 | Trivy | 容器镜像扫描 |
| 7 | 单元测试 | UT 覆盖率 | 覆盖率是否达标 |

#### 6.2 自动修复循环

```
while ROUND ≤ MAX_FIX_ROUNDS:
    run_gates()
    if PASS → break
    if 最后一轮 → break
    if 不可自动修复(license/design) → break
    auto_fix(ROUND)
    ROUND += 1
```

三类检查的修复策略：
- **可自动修复**：敏感信息（删除密钥）、依赖漏洞（升级版本）→ dev Agent 收到反馈后改代码
- **不可自动修复**：许可证合规、设计文档缺失 → 直接失败，留给人工处理
- **仅检查不修**：部分项目不适用 → 跳过

#### 6.3 Security Gate 预跑（`security_gate_local.sh`）

在 Phase B2 的 submit 段，**开 PR 之前**本地预跑 Security Gate：

| 模式 | 触发方式 | 行为 |
|------|---------|------|
| 默认（best-effort） | `/ai-develop-submit` | 只跑 Gitleaks（敏感信息） |
| 全量模式 | `/ai-develop-submit --allgate` | 跑全部 6 项 |

设计意图：**安全门禁前移**——在 PR 创建之前就发现问题并修复，避免 PR 提交后当场被 CI 挂红。如果预跑失败，dev Agent 会修复 → 重跑，最多 `MAX_FIX_ROUNDS` 轮。

#### 6.4 Gate 在流程中的位置

```
Phase B2 submit 段:
  deploy → gates/run.sh → review → tester → security-gate预跑 → 开PR
              │                                            │
              └── 7 项门禁检查                              └── PR 前的安全检查
              └── 失败 → auto_fix → 重跑
```

## 7. runtime-clone（运行时克隆部署模式）

### 通用定义

**runtime-clone** 是 backlog `.ai-flow` 默认的预览部署模式。它不是用 Dockerfile 构建镜像再部署，而是：
1. 用一个**干净的通用基础镜像**（如 `python:3.12`）启动 Pod
2. Pod 启动后 **现场从 GitHub `git clone` 代码**
3. 安装依赖、编译、运行
4. 服务对外暴露

### 在 backlog 中的具体含义 (+)

```
Pod 启动脚本（简化）：
  git clone --depth 1 --branch issue-N-impl <repo-url> /tmp/app
  cd /tmp/app
  pip install --no-cache-dir -r requirements.txt   # 或 npm ci / go build
  exec python main.py                               # 或 npm run dev / go run
```

#### 与 Dockerfile 模式对比

| 维度 | runtime-clone | Dockerfile 构建 |
|------|--------------|----------------|
| 镜像构建 | 不需要 | 需要 Docker build + push |
| 部署速度 | 快（秒级 clone） | 慢（分钟级 build + push） |
| 环境一致性 | 基础镜像统一，但依赖安装时机不同 | 镜像即环境，高度一致 |
| 适用场景 | **预览/开发**环境，快速迭代 | **生产**环境，可重复部署 |
| 缺陷 | 预览环境 ≠ 生产环境（语言基础镜像可能缺少生产工具） | — |

#### 语言自动探测

backlog 的 `deploy.sh` 按以下规则选基础镜像：

```
go.mod         → golang:1.22
package.json   → node:20
*.py           → python:3.12
其他            → python:3.12（默认）
```

#### 为什么要 runtime-clone？

预览环境的本质需求是"**快速、低成本地看到一个分支的运行效果**"。如果每次预览都要走完整镜像构建流程（Docker build → push → deploy），预览周期将从**分钟级膨胀到十分钟级**，失去"预览"的意义。

## 8. Secret 烘培（Secret Baking）

### 通用定义

"烘培"（Baking）借用了烤面包的比喻：**把散乱的食材（配置片段）通过一套工艺加工成可以直接摆上桌的成品（k8s Secret）**。

在 K8s 语境中，它指从不同来源（Vault、环境变量、配置文件）收集配置信息，经过转换/改写后，做成 k8s Secret 资源。

### 在 backlog 中的具体含义 (+)

以 forum-reply-robot 的 `sync.sh` 为例，完整烘培流程：

```
① 读 service.md
   → 找到服务的 Vault Path

② 登录 Vault（userpass 认证）
   → 获得临时 VTOKEN

③ curl 读 Vault Path
   → 获取 JSON: {"data":{"data":{"config":"...","secrets":"..."}}}

④ Python 解析 + 改写
   改写的关键内容：
   ┌──────────────────────────────────────────────┐
   │  数据库 host  → postgresql-service.<ns>.svc   │
   │            port → 5432                         │
   │            user → postgres                     │
   │         password → 预览底座 PG 密码             │
   │         database → forum_reply_robot           │
   │                                                │
   │  域名          .test.osinfra.cn                │
   │              → .preview.test.osinfra.cn        │
   │                                                │
   │  DEBUG 等标志  False → True (预览模式)          │
   └──────────────────────────────────────────────┘

⑤ kubectl create secret generic <name>-config
      --from-file=config=<改写后的config>
      --from-file=secrets.yaml=<改写后的secrets>

⑥ 挂载到 Pod
   /vault/secrets/config  → cp 到 config/config.yaml → 应用启动
```

### 为什么需要烘培？

**生产环境的配置不能直接用于预览环境**：
- 生产数据库地址 → 需要指向预览底座 PostgreSQL
- 生产域名 → 需要改成预览域名
- 生产 API Key → 可能需要用测试凭证

因此需要一套"取真配置 → 改写为预览形态 → 注入 Pod"的流程，这就是 Secret 烘培。

### mapping 字典的作用

```python
# forum-reply-robot 的 mapping
mapping = {"config": "config", "secrets": "secrets.yaml"}
```

定义了 Vault Key → Pod 内文件名的映射。例如 Vault 中的 `config` key 内容写入文件 `config`，挂载后 Pod 内路径为 `/vault/secrets/config`。

## 9. ArgoCD（GitOps 工具）

### 通用定义

ArgoCD 是一个 **GitOps 持续交付工具**。它的核心逻辑是：

```
你的 Git 仓库里存着 K8s YAML 文件（期望状态）
        │
        ▼
ArgoCD 每隔 3 分钟检查 Git 仓库是否变了
        │
        ▼
如果有变更 → 自动同步到 K8s 集群（使实际状态 = 期望状态）
        │
        ▼
如果 K8s 里有人手动改了配置 → ArgoCD 检测到偏离 → 自动回滚到 Git 版本
```

一句话：**Git 是唯一真相源，ArgoCD 确保 K8s 永远和 Git 一致**。

### 在 opensourceways 中的角色

```
CI 流水线（Phase C 发布）：
  构建镜像 → 推到 SWR/Harbor
           → 改 GitOps 仓的镜像 tag
                 │
                 ▼
           infra-common 或 helm-chart-value 仓库
           （kustomize YAML 或 values.yaml 中的 .image.tag 字段）
                 │
                 ▼
           ArgoCD 检测到 Git 变更
                 │
                 ▼
           自动同步到 K8s 集群
                 │
                 ▼
           新版本上线
```

### 关键字段（在 infrastructure/service.md 中）

| 字段 | 示例 | 用途 |
|------|------|------|
| ArgoCD 域名 | `https://build-01.test.osinfra.cn` | ArgoCD 管理界面地址 |
| ArgoCD 应用名 | 各服务注册的应用名 | 在 ArgoCD 中唯一标识一个应用 |
| 镜像版本路径 | `.image.tag` 或 `.spec.template.spec.containers[0].image` | CI 发布时改写的字段位置 |

### ArgoCD 与 runtime-clone 的关系

| | runtime-clone | ArgoCD |
|------|-------------|--------|
| 使用阶段 | Phase B（预览/开发） | Phase C（发布/上线） |
| Git 角色 | 代码源（clone 后编译运行） | 部署配置源（只存 YAML） |
| 镜像 | 不需要镜像 | 需要预构建的 Docker 镜像 |
| 环境 | 临时预览环境 | 稳定的 test / prod 环境 |

## 10. GitOps

### 通用定义

GitOps 是一套**用 Git 驱动运维操作**的方法论，由 Weaveworks 于 2017 年提出，核心原则：

1. **Git 是唯一真相源**：所有 K8s 配置（Deployment/Service/ConfigMap/Secret）存在 Git 仓库中
2. **声明式配置**：不写 `kubectl run` 命令，而是写 YAML 描述"应该是什么样"
3. **自动同步**：用 Operator（如 ArgoCD/FluxCD）持续将 Git 状态同步到集群
4. **闭环比对**：Operator 持续监控实际状态与 Git 声明的差异

### GitOps vs 传统 CI-Ops

| 维度 | 传统 CI-Ops | GitOps |
|------|-----------|--------|
| 部署触发 | CI 脚本执行 `kubectl apply` | Git 仓库变更 → Operator 自动同步 |
| 权限模型 | CI 系统有集群写权限 | 只有 Operator 有集群写权限 |
| 回滚方式 | `kubectl rollout undo` | `git revert` |
| 审计日志 | CI 运行历史 | Git commit log |
| 集群环境 | CI runner 需要能访问 K8s | Operator 在集群内运行 |

### 在 opensourceways 中的体现

```
GitOps 仓库: infra-common / helm-chart-value
    ├── 存放所有服务的 K8s YAML（Deployment/Service/Ingress 等）
    ├── 按 <项目>/<服务>/<环境> 目录层级组织
    └── 镜像版本通过 kustomize 或 Helm values.yaml 管理

GitOps 引擎: ArgoCD
    ├── 监控 GitOps 仓库变更
    ├── 自动同步到 K8s 集群
    └── 偏离时自动回滚

发布流程:
    AI 构建镜像 → 推 SWR → 改 infra-common 的镜像 tag → ArgoCD 同步
```

## 11. GitHub Actions CronJob（定时任务）

### 通用定义

GitHub Actions 支持通过 `schedule` 事件触发 Workflow 以 **cron 表达式**定时执行：

```yaml
# .github/workflows/weekly-eval.yml
on:
  schedule:
    - cron: '0 0 * * 1'  # 每周一 UTC 00:00 执行
```

这相当于在 CI 体系中跑 **cron job**，无需额外服务器。

### 在 backlog / forum-reply-robot 中的用途 (+)

在 `forum-reply-robot-issue分析.md` 的阶段三设计中，CronJob 被用于**RAG 质量持续看护**：

```
每周一 00:00 UTC 定时执行：

1. 从最近一周新问题中分层抽样补充评估集（每类 +10 条）
2. 对全量评估集并行执行两条管道：
   - LightRAG 管道：读取已记录的 retrieval_context + actual_output
   - pgvector 管道：运行影子管道产出新的结果
3. deepeval 分别对两条结果打分
4. 产出对比报告：
   - Contextual Relevancy 趋势图（LightRAG vs pgvector）
   - Faithfulness 趋势图
   - 配对 t 检验结果（p < 0.05）
   - 差异最大 top-10 case
5. 报告存储到 PostgreSQL + 生成 Markdown 摘要评论到对应 Issue
```

### CronJob vs Event-Driven

| 维度 | CronJob（定时） | Event-Driven（事件驱动） |
|------|--------------|---------------------|
| 触发方式 | 到点自动执行 | Issue 评论/标签变化时触发 |
| 适用场景 | 定期看护、周期性巡检 | 开发流水线（预览/提交/发布） |
| 在 backlog 中 | **RAG 回归评测**（每周） | **所有开发流程**（command-router） |
| 延迟 | 最长可差一个定时间隔 | 实时 |

### GitHub Actions Cron 的局限

- **最细粒度 5 分钟**（不能"每 30 秒"执行）
- **不保证准时**：高负载时可能延迟最多几十分钟
- **仓库 60 天无活动则自动停用** schedule Workflow
- 公开仓库免费额度充裕（2000 分钟/月），私有仓库有限

## 12. SWR（华为云容器镜像服务）

### 全称

**SoftWare Repository for Container** — 华为云容器镜像服务。

### 是什么

SWR 是华为云提供的 **Docker 镜像仓库**，对标 Docker Hub / Harbor / AWS ECR。用来存储、分发、管理容器镜像。

### 在 opensourceways 中的角色

opensourceways 的**所有服务镜像都推送到 SWR**，地址规律：

```
swr.cn-north-4.myhuaweicloud.com/<组织名>/<镜像名>:<tag>
                                ↑              ↑
                            opensourceway    forum-robot
                          （企业版）或
                            openubmc
```

### 完整发布链路

```
forum-reply-robot 代码仓库
    │
    ▼
Jenkins pipeline（infrastructure/pipeline/discourse-forumRobot.txt）
    │  docker build -t forum-robot:v1.0.xxx .    ← 用 Dockerfile 构建
    │  docker tag → swr.cn-north-4.../opensourceway/forum-robot:v1.0.xxx
    │  docker push                                 ← 推到 SWR
    ▼
SWR（华为云镜像仓库）
    │  swr.cn-north-4.myhuaweicloud.com/opensourceway/forum-robot:v1.0.xxx
    ▼
GitOps 仓（helm-chart-value）更新 tag → ArgoCD 检测 → 拉 SWR 镜像 → 部署到 K8s
```

### 两层 SWR 推送

Jenkins pipeline 实际推到**两个组织**下：

```bash
# 第一层：opensourceway 组织（主要镜像仓）
docker push swr.cn-north-4.../opensourceway/forum-robot:v1.0.xxx

# 第二层：openubmc 组织（备份/多租户镜像仓）
docker tag  → swr.cn-north-4.../openubmc/forum-robot:v1.0.xxx
docker push
```

### SWR vs Docker Hub vs sync.sh

| | SWR 正式镜像 | sync.sh 预览 Pod |
|---|---|---|
| 构建方式 | Dockerfile `docker build` | `python:3.9-slim` + 现场 `pip install` |
| 构建时间 | 分钟级 | 秒级 `git clone` |
| 内容完整性 | 完整（含 SchemaFiles、安全加固、目录创建） | **不完整**（缺 SchemaFiles 等 Dockerfile 步骤） |
| 使用阶段 | Phase C 测试发布 / 正式上线 | Phase B 预览 |
| 存储位置 | 华为云 SWR | 不存储（临时 Pod） |

### 关键认知

- SWR 只是一个**存储+分发中心**，不参与构建
- 镜像内容由 **Dockerfile** 定义，Jenkins pipeline 只是调 `docker build` + `push`
- preview 和 test/prod 的镜像来源不同：preview 不走 SWR（现场克隆），test/prod 走 SWR（预构建完整镜像）
- 这就是为什么**预览 Pod 的行为可能与测试环境不同**——预览缺少 Dockerfile 里的额外构建步骤
