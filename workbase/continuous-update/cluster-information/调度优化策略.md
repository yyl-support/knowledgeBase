# 调度优化策略

**环境**：正式贵阳 Karmada 联邦（控制面 1.95.223.182:5443，成员集群 001 + wlcb）
**关联文档**：`正式环境集群全景.md` · `集群对比_001_vs_wlcb.md` · `22点弹ECS诊断报告.md`

---

## 一、问题

### 1. 22:00 集中弹 ECS（每天重复）

```
ascend-infra 流水线 22:00 批量提交 CI 作业
  → Argo parallelism 无限 → 瞬间触发近百个 VCJob
  → Volcano queue 无 capability → 作业全部放行
  → Karmada p10 兜底策略 → 分发到 001 + wlcb
  → wlcb 被 NPU 训练任务占满（64 张卡）
  → estimator 把全部作业塞进 001
  → 001 CPU/内存超 80%
  → HNA 逐台弹 ECS（+1，无冷却，无上限）
  → 一次高峰弹 20+ 台
```

**核心矛盾**：不是 001 资源不够，是涌入太猛、wlcb 又吃不下，Karmada 被迫把所有作业倒进一个集群。

### 2. clusterNames 硬编码

所有分发策略写死 `clusterNames: ["001","wlcb"]`，新增集群要改十几条策略。

---

## 二、方案

按问题位置分成四段，对应数据流的四个环节：

```
[CodeArts] → [Karmada 分发] → [集群内调度] → [弹性伸缩]
    ↑              ↑               ↑              ↑
  流水线限流     去硬编码         装箱偏好        HNA 刹车
```

### A. 流水线限流（根源，最优先）

**改什么**：Argo Workflow 模板 + Volcano 队列配置。**不改 Karmada，不改集群**。

1. Argo Workflow 加 `parallelism` 上限（建议 5-10）：
   ```
   之前：22:00 瞬间 100 个 VCJob 同时创建
   之后：排队，同时最多 5-10 个跑
   ```

2. Volcano 队列加 `capability` 封顶：
   ```yaml
   spec:
     capability:
       cpu: "2000"
       memory: "8Ti"
   ```
   超过 cap 的作业在队列里排队，不创建 Pod。001 不可能被撑爆。

   > ✅ 已核实：[Volcano Queue 官方文档](https://volcano.sh/en/docs/Concepts/Queue) 支持 `capability` 字段，格式为资源键值对（非列表）。

**效果**：22:00 弹 ECS < 5 台。这是唯一能根治的措施，其余都只能减轻。

---

### B. Karmada 分发：去 clusterNames 硬编码

#### B1. 集群标签

> ✅ 已验证：`clusterAffinity.labelSelector` 在生产 Karmada（服务端 v1.34.1）上 dry-run 通过，语法可用。

| 集群 | 标签 | 含义 |
|------|------|------|
| 001 | `dispatch/auto=true`, `cluster-role=cpu,npu-b` | 主力混合集群 |
| wlcb | `dispatch/auto=true`, `cluster-role=npu-a` | NPU 专用集群 |

#### B2. Job 分发策略（4 条）

| 策略 | 匹配对象 | placement 改后 |
|------|----------|---------------|
| p100 (2条) | `dispatch/001=true` / `dispatch/wlcb=true` | 不改（人工钉死） |
| p20（非 NPU） | 无 NPU 标签的 VCJob | `labelSelector: cluster-role=cpu` |
| p10（兜底） | 其余所有 VCJob | `labelSelector: dispatch/auto=true` |

改法：每条策略 `placement.clusterAffinity.clusterNames` → `labelSelector`。

#### B3. Namespace 下发（11 条）

只改 placement，不改数量，不改 resourceSelectors：

```yaml
# 改前
clusterAffinity:
  clusterNames: ["001","wlcb"]

# 改后
clusterAffinity:
  labelSelector:
    matchLabels:
      dispatch/auto: "true"
```

11 条策略每条独立管控自己的 ns，placement 统一用 labelSelector。新增集群打标即生效，某个 ns 要定制目标集群改自己那条就行。

#### B4. 补齐 + 兜底

- `volcano-global-all-queue-propagation` 和 `volcano-global-dispatch-policy` 补全 `clusterAffinity`
- 新增 catch-all 策略 + 巡检脚本（30 分钟扫一次 ResourceBinding 为空）

---

### C. 集群内调度：pre-paid 优先装箱

**问题**：Volcano `leastrequested/mostrequested=0`，CPU 作业在固定机和 ECS 之间平权离散散布，固定机空着也弹 ECS。

**方案**：扩展 `non-npu-vcjob-node-affinity` cop，新增 pre-paid 软偏好：

```yaml
preferredDuringSchedulingIgnoredDuringExecution:
- weight: 100
  preference:
    matchExpressions:
    - key: node.cce.io/billing-mode
      operator: In
      values: ["pre-paid"]
```

Weight 100 × nodeaffinity.weight 2000 → 固定机有空位必落固定机。固定机满了才落 ECS，不阻塞。

**为什么用 cop 而不是改 scheduler config**：改全局 scheduler 影响所有工作负载（含 NPU、常驻服务），风险太大。cop 精确命中"非 NPU 的 Volcano Job"。

---

### D. 弹性兜底：HNA 加刹车

| 参数 | 现状 | 改后 |
|------|------|------|
| 触发阈值 | CPU/内存 > 80% | **85–90%** |
| 扩容步长 | +1 | **+2 或 +3** |
| 冷却窗口 | 无 | **≥5 分钟** |
| 最大节点数 | 无上限 | **设 max** |

A、B、C 三段做完后 HNA 触发次数应大幅下降，HNA 调优只是收尾。

---

## 三、实施顺序

| 阶段 | 内容 | 依赖 |
|------|------|------|
| **1. 采集基线** | 22:00 前后四个维度的数据（队列状态 / RB 落点 / 节点分布 / HNA 记录） | — |
| **2. 流水线限流** | Argo parallelism + 队列 capability | 无，可独立上线 |
| **3. Karmada 分发** | 集群打标 → 改 cpp 为 labelSelector → 补齐 + catch-all | 需先确认群标签 |
| **4. 集群内调度** | 扩展 cop 加 pre-paid 软偏好 | Phase 3 之后 |
| **5. 弹性刹车** | HNA 灰度调参（阈值 → 冷却 → max，每次一项） | Phase 4 之后 |
| **6. 效果验证** | 对比基线，达标则收尾 | Phase 5 全部完成 |

每阶段改完观察 1-2 个 22:00 周期再进入下一阶段。每步改动前 `kubectl get <资源> -o yaml > backup-$(date +%F)-<名>.yaml`。

---

## 四、回滚

按影响面从大到小：**先回弹性 → 再回调度 → 再回分发 → 最后回限流**。

- **队列 cap**：`kubectl edit queue <q>` 删 spec.capability
- **cpp/cop**：`kubectl apply -f backup-<日期>-<名>.yaml`
- **集群标签**：`kubectl label cluster <名> <key>-`
- **HNA**：`kubectl apply -f backup-<日期>-hna.yaml`
