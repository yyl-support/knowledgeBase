## 一、集群全景（共 5 个集群）

> kubeconfig 里的 `prod-guiyang-ipv6-internal/-external/-externalTLSVerify` 是**同一个集群**的三种访问入口（内网 / 外网 / 外网+校验证书），不是三个集群。

| 集群 (context)                  | 角色                            | K8s 版本       | 规模          | 定位                                                          |
| ----------------------------- | ----------------------------- | ------------ | ----------- | ----------------------------------------------------------- |
| `prod-guiyang-karmada`        | **Karmada 控制面**（联邦 apiserver） | v1.31        | 逻辑面，无业务节点   | 存放 pp/cpp/op/cop 策略，决定作业下发到哪个成员集群                           |
| `prod-guiyang-karmada-deploy` | **Karmada 宿主/部署集群**           | v1.33（较新）    | 2 节点（小）     | 承载 karmada 组件：etcd×3、apiserver、scheduler、operator、estimator |
| `prod-0001`（成员 `001`）         | **主力混合成员集群**                  | v1.31.14     | 58 节点       | NPU(910b/310p)+CPU+弹性 ECS，承接非 NPU 批作业与潮汐负载                  |
| `prod-wlcb`（成员 `wlcb`）        | **NPU 专用成员集群**                | v1.31.10（略旧） | 11 节点       | 固定物理 910b(313t)，几乎不弹性                                       |
| `prod-guiyang-ipv6`           | **独立新集群**（非 Karmada 成员）       | v1.35（最新）    | 3 节点（纯 CPU） | IPv6、独立 argo/harbor，像新环境/试点，自成一体                            |

### 1.1 两套体系

**① Karmada 联邦体系（4 个集群协同）**

```
prod-guiyang-karmada-deploy   （宿主：装着 Karmada 组件）
        └── prod-guiyang-karmada   （控制面：联邦大脑，做下发决策）
                ├── 001    （成员：主力弹性，跑非 NPU 批作业 + 部分 NPU）
                └── wlcb   （成员：NPU 专用固定池）
```

- **控制面 `prod-guiyang-karmada`**：只做“把作业派到哪个集群”的决策，本身不跑业务。所有传播/改写策略（pp/cpp/op/cop）都在这里。
- **宿主 `prod-guiyang-karmada-deploy`**：控制面的物理载体。`karmada-system` 里可见 `ascend-ci-karmada-apiserver`、3 副本 `etcd`、`karmada-scheduler`、`karmada-operator`，以及为两成员估算容量的 `karmada-scheduler-estimator-001` / `-wlcb`。
- **成员 `001` / `wlcb`**：真正执行作业的集群（详见 `集群对比_001_vs_wlcb.md`）。

**② 独立集群 `prod-guiyang-ipv6`**

- 不在 Karmada 成员列表中（控制面只管 001/wlcb），**独立自治**。
- IPv6、K8s v1.35 最新、仅 3 个纯 CPU 节点、Age ~23 天，命名空间仅 argo/harbor/git-cache/monitoring 等基础组件。判断为**新搭的独立/试点环境**，与联邦体系当前无调度关系。

## 二、调度分配：任务进来后凭什么分到 001 还是 wlcb

分配发生在 **Karmada 控制面**，由多条 `ClusterPropagationPolicy`（cpp）按 **priority 从高到低**竞争匹配：**资源命中的、优先级最高的那条策略的 `placement` 决定它去哪个集群**。

### 2.1 规则总表（按优先级）

| 优先级     | 策略                            | 命中条件（resourceSelector）                                | 落到哪                 |
| ------- | ----------------------------- | ----------------------------------------------------- | ------------------- |
| **100** | `member1-pod-policy`          | 标签 `dispatch/001=true`                                | **001**（强制钉住）       |
| **100** | `wlcb-pod-policy`             | 标签 `dispatch/wlcb=true`                               | **wlcb**（强制钉住）      |
| **20**  | `non-npu-vcjob-prefer-001`    | Job/vcjob 且 `huawei.com/npu` **不存在**（非 NPU 作业）        | **001**             |
| **10**  | `default-job-dispatch-policy` | Job/vcjob 且 **无** `dispatch/001` 和 `dispatch/wlcb` 标签 | **001 + wlcb 之间分配** |

### 2.2 判定流程（一个作业进来的走向）

```
作业(Job / Volcano vcjob) 提交
        │
        ├─ 带标签 dispatch/001=true ? ──是→ 强制到 001         (member1-pod-policy, p100)
        │
        ├─ 带标签 dispatch/wlcb=true ? ──是→ 强制到 wlcb        (wlcb-pod-policy,   p100)
        │
        ├─ 是「非 NPU」作业(无 huawei.com/npu)? ──是→ 到 001    (non-npu-vcjob-prefer-001, p20)
        │        └─ 落到 001 后，再被 override 注入 nodeAffinity：
        │           只上非 NPU 节点（并按方案将“软优先固定机”）
        │
        └─ 其余(通常是 NPU 作业，未显式钉集群) ─→ 001 与 wlcb 之间分配
                 (default-job-dispatch-policy, p10)
                 replicaScheduling: Divided + Aggregated
                 由 karmada-scheduler + estimator(001/wlcb) 按可用容量决定
```

### 2.3 关键点解读

1. **显式钉住优先级最高**：只要作业打了 `dispatch/001=true` 或 `dispatch/wlcb=true`（priority 100），无条件去对应集群，覆盖其它规则。这是运维/流水线定向投递的手段。
2. **非 NPU 作业固定去 001**（priority 20）：凡是不带 NPU 资源标签的 Job/vcjob，一律派到 001。这正是 22:00 大批 CI 构建作业压向 001、进而弹 ECS 的分配根源。
   - 注：该策略线上 `spec` 目前 `clusterAffinity` 只写了 `001`（无 wlcb 兜底）；其 `last-applied` 注解里曾是 `clusterAffinities`（prefer 001 → fallback wlcb），说明被改成了“只钉 001”。
3. **其余作业在两集群间分配**（priority 10，兜底）：既没显式钉、又是 NPU 作业的，走 `default-job-dispatch-policy`，Karmada 用 `Divided + Aggregated` 策略把副本在 001/wlcb 间划分，并借助 `scheduler-estimator-001/-wlcb` 判断各成员的可用容量再决定。
4. **两级调度**：Karmada 决定“去哪个**集群**”（集群间）；作业落地后，集群内的 **Volcano 调度器**再决定“上哪台**节点**”（节点间，详见 `调度优化方案_固定机优先.md`）。

## 三、复核命令

```bash
# 成员集群列表与状态
kubectl --context prod-guiyang-karmada get clusters -o wide

# 分配相关策略（按优先级看）
kubectl --context prod-guiyang-karmada get cpp
kubectl --context prod-guiyang-karmada get cpp default-job-dispatch-policy -o yaml
kubectl --context prod-guiyang-karmada get cpp non-npu-vcjob-prefer-001    -o yaml
kubectl --context prod-guiyang-karmada get cpp wlcb-pod-policy             -o yaml
kubectl --context prod-guiyang-karmada get cpp member1-pod-policy          -o yaml

# 某作业实际被派到哪些集群（调度结果）
kubectl --context prod-guiyang-karmada get rb -A        # ResourceBinding

# Karmada 组件（宿主集群）
kubectl --context prod-guiyang-karmada-deploy get pods -n karmada-system

# 独立 ipv6 集群
kubectl --context prod-guiyang-ipv6-external get nodes
```
