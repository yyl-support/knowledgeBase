## 一、问题全景

### 1.1 现象

- **A**：22:00 前后，001 集群批量弹出 20+ 台 ECS（每天重复）
- **B**：策略系统 `clusterNames` 硬编码，新增集群需改 13 处
- **C**：NPU 算力主要集中在 001（80 张），wlcb 排程基本满载，CI 任务很难分到 wlcb

### 1.2 真实根因链（与 22:00 弹 ECS 对应）

```
[源头] ascend-infra 流水线 22:00 整点批量提交 CI 作业
   │  Argo parallelism 没限 → 同一时刻触发数十个 Workflow
   │  Volcano queue capability 没设 → 作业无上限并发
   ▼
[分发] Karmada p10 default-job-dispatch-policy 兜底分到 001+wlcb
   │  这些是 ascend-* 作业，带 huawei.com/npu 标签 → 跳过 p20（非 NPU）
   │  没带 dispatch label → 跳过 p100
   │  走 p10: clusterNames=["001","wlcb"], Divided+Aggregated
   │  estimator 看到 wlcb 已被 NPU 训练任务占满（64 张卡）
   │  → 全部涌入 001
   ▼
[节点选择] Volcano Scheduler
   │  leastrequested/mostrequested=0
   │  binpack 只对 ascend-1980/310 生效
   │  → 作业离散散布，pre-paid/post-paid 平权
   ▼
[弹性] 001 CPU/内存 > 80% → HNA 触发
   │  每次 +1，无冷却，无上限
   │  一次高峰 → 几十次扩容 → 20+ ECS
   ▼
[副作用] 高峰期部分 AZ ECS 规格库存不足 → 跨规格重试 → 扩容噪音 + 失败事件
```

**关键纠正**：22:00 弹 ECS 的作业是 **ascend-* NPU CI 任务**（不是非 NPU 任务），它们走 **p10 兜底策略**（不是 p20），主要因为 wlcb 已被专用 NPU 训练占满、Karmada estimator 把它们全部分到了 001。

### 1.3 三个共犯 + 一个根因

| 层级 | 当前做法 | 角色 |
|------|----------|------|
| 源头（Argo / 队列） | parallelism 不限、capability 不设 | **真正的根因** |
| 分发（Karmada） | `clusterNames` 硬编码、单点汇聚 | **传导环节** |
| 调度（Volcano Scheduler） | 无装箱偏好 | **放大效应** |
| 弹性（HNA） | 利用率触发、+1、无冷却、无上限 | **放大效应** |

只有源头（Argo / 队列）解决才能根治。其它三层只能"减轻症状"。

## 二、方案架构：四层治理

```
第 0 层：源头削峰     → 治本
第 1 层：Karmada 分发 → 治标（分散负载）
第 2 层：集群内调度   → 治标（装箱偏好）
第 3 层：弹性兜底     → 治标（HNA 刹车）
```

**第 0 层必须先上**，否则后三层只能治标不能治本。第 0 层可独立落地，不需要改其它任何东西。

## 三、第 0 层：源头削峰（治本）

### 3.1 Argo parallelism 限制

`ascend-infra` 仓库的 Workflow 模板里设置 `parallelism` 上限（如 5–10），控制同时跑的 step 数量。

**效果**：22:00 批次原本 100 个 vcjob 同一时刻涌入，限制后 5–10 个同时跑，其余排队。

### 3.2 Volcano 队列 capability 封顶

```bash
# kubectl --context prod-0001
kubectl edit queue large-task-shared-queue
kubectl edit queue shared-flexible-queue
```

在队列 spec 里加 `capability`（CPU / 内存 / 任务数）。以任务数封顶最简单：

```yaml
spec:
  capability:
    - name: tasks
      value: "30"      # 同时最多 30 个 vcjob
    - name: cpu
      value: "2000"    # 2000 核
    - name: memory
      value: "8Ti"     # 8T 内存
```

**效果**：超过 cap 的作业在队列里排队而非直接调度。22:00 涌入再多也只占 cap 内的资源，绝不超 001 节点容量。

### 3.3 验证

- 22:00 周期 001 CPU/内存峰值 < 70%（cap 内运行的安全余量）
- 队列里 `Pending` 作业数 > 0（说明 cap 生效在排队）
- 弹 ECS 台数 < 5（之前 20+）

## 四、第 1 层：Karmada 分发优化

### 4.1 集群标签设计

仅 001 和 wlcb 两个成员集群需要打标签：

| 集群 | `cluster-role` | `dispatch/auto` | 备注 |
|------|---------------|-----------------|------|
| 001 | `cpu,npu-b` | `true` | CPU 主力 + 异构 NPU（80 张卡，比 wlcb 多） |
| wlcb | `npu-a` | `true` | NPU 专用精简集群（64 张卡） |

- `npu-a` = NPU 专用集群，`npu-b` = 含 NPU 的混合集群，`cpu` = 纯 CPU/弹性池
- `cluster-tier` 移除——pre-paid/post-paid 是 node 属性不是 cluster 属性，由第 2 层处理

### 4.2 分配类策略（4 条 + 1 个 pp）

**p100 钉死策略（2 条）：不改。** `dispatch/001=true` / `dispatch/wlcb=true` 是人工显式钉集群，语义不需要泛化。

**p20 非 NPU 分发：标签化 + 改名**：

```yaml
# 改前
placement:
  clusterAffinity:
    clusterNames: ["001"]
# 改后（policy 名改为 non-npu-vcjob-cpu-prefer）
placement:
  clusterAffinity:
    labelSelector:
      matchLabels:
        cluster-role: cpu
```

**p10 兜底分发：标签化**：

```yaml
# 改前
placement:
  clusterAffinity:
    clusterNames: ["001","wlcb"]
# 改后
placement:
  clusterAffinity:
    labelSelector:
      matchLabels:
        dispatch/auto: "true"
```

> p10 改动对当前 22:00 行为**无影响**——22:00 作业走的就是 p10，新策略仍匹配 001+wlcb，匹配范围不变。改动只对"未来新增 CPU 集群"有意义。

**pp 命名空间级策略（`argo-npuir-policy`, p20）：保留。** argo ns 下 `jobRepositoryName=ascend-ascendnpu-ir` 钉到 wlcb，本身就是合理的 ns 级调度，无需泛化。

### 4.3 命名空间下发（11 条 → 1 条）

```yaml
kind: ClusterPropagationPolicy
metadata:
  name: namespace-duplicate-to-all
spec:
  priority: 0
  placement:
    clusterAffinity:
      labelSelector:
        matchLabels:
          dispatch/auto: "true"
    replicaScheduling:
      replicaSchedulingType: Duplicated
  resourceSelectors:
  - apiVersion: v1
    kind: Namespace
    labelSelector:
      matchLabels:
        karmada.io/managed: "true"
```

现有 11 个 ns 全部打 `karmada.io/managed=true` 标签后，自动复制到 `dispatch/auto=true` 的所有成员集群。

**11 → 1，新增集群零改动。**

### 4.4 补全缺失的 clusterAffinity

```yaml
# volcano-global-all-queue-propagation (cpp, p0)
# volcano-global-dispatch-policy (pp, default ns, p0)
# 补全：
placement:
  clusterAffinity:
    labelSelector:
      matchLabels:
        dispatch/auto: "true"
```

### 4.5 catch-all 兜底

新增 p0 全匹配 cpp + 巡检脚本（30 分钟扫一次 ResourceBinding 为空的条目）。

```yaml
kind: ClusterPropagationPolicy
metadata:
  name: catch-all-no-match
spec:
  priority: 0
  placement:
    clusterAffinity:
      labelSelector:
        matchLabels:
          dispatch/auto: "true"
  resourceSelectors:
  - apiVersion: batch.volcano.sh/v1alpha1
    kind: Job
  - apiVersion: batch/v1
    kind: Job
```

> Karmada 原生不支持在策略匹配时打 event。catch-all 实际靠巡检脚本告警。

## 五、第 2 层：集群内调度优化

### 5.1 根因

Volcano `nodeorder` 配置：

```
nodeaffinity.weight:    2000
leastrequested.weight:  0       ← 不鼓励装箱
mostrequested.weight:   0       ← 不鼓励装箱
binpack.weight:         1       ← 仅 ascend-1980/310，不管 CPU
```

CPU 作业在 pre-paid 固定机和 post-paid ECS 间**平权打分，离散散布**。

### 5.2 方案：cop 注入 pre-paid 软偏好

扩展 `ClusterOverridePolicy/non-npu-vcjob-node-affinity`：

```yaml
nodeAffinity:
  # 保留：硬约束——非 NPU 作业不上 NPU 节点
  requiredDuringSchedulingIgnoredDuringExecution:
    nodeSelectorTerms:
    - matchExpressions:
      - key: accelerator/huawei-npu
        operator: DoesNotExist
  # 新增：软偏好——优先填满 pre-paid 固定机
  preferredDuringSchedulingIgnoredDuringExecution:
  - weight: 100
    preference:
      matchExpressions:
      - key: node.cce.io/billing-mode
        operator: In
        values: ["pre-paid"]
```

**生效原理**：soft preference（weight=100）被 `nodeaffinity.weight=2000` 放大 → pre-paid 打分远超 post-paid → 固定机有空位必落固定机；固定机满了才落 ECS（不阻塞）。

**为什么用 cop 而不是改 scheduler config**：`leastrequested/mostrequested` 改动影响全部工作负载（含 NPU、常驻服务），风险面不可控。cop 精确命中"非 NPU 的 Volcano Job"。

### 5.3 跨策略一致性

`member1-secret-policy` 和 `member2-secret-policy` 也用 `dispatch/001=true` / `dispatch/wlcb=true` 标签。Phase 2 改完 cpp 后，dispatch label 对 Secret 和 Job 行为一致——Secret 仍按 `dispatch/001=true` 钉死，Job 走 cpp 的两条 p100 钉死策略。无冲突。

## 六、第 3 层：弹性兜底

### 6.1 HNA 参数收敛

即使前 3 层都做了，HNA 仍是最后一道闸门。要给它加刹车：

| 参数 | 现状 | 改后 | 理由 |
|------|------|------|------|
| 触发阈值 | CPU/内存 > 80% | **85–90%** | 给固定机留更多填满空间 |
| 扩容步长 | 每次 +1 台 | **+2 或 +3** | 减少触发次数 |
| 冷却窗口 | 无 | **≥5 分钟** | 新 ECS 就绪需 3-5 分钟，冷却期内不重复触发 |
| 最大节点数 | 无上限 | **设 max** | 成本封顶，按历史峰值留余量 |

> **HNA 是利用率触发，无法做到"Pending 才弹"**——但前 3 层做好了，假阳性会自然减少。HNA 调优仍是配套收尾，不是根治。

### 6.2 ECS 规格库存

诊断报告附录指出 `policy-1780919153912` 节点池在高峰期 `kc2.32xlarge.4 / kc2.20xlarge.4` 规格库存不足，触发跨组重试。

**需核对**：节点池可用区配置、规格配比。备选规格补齐后减少 `No available flavors` 失败重试。

## 七、技术债务

| 债务 | 说明 | 建议 |
|------|------|------|
| cop 路径硬编码 `/spec/tasks[0]` | 多 task 作业只有首个 task 被注入 | 下沉到 converter，在生成 YAML 时直接写 affinity |
| HNA 利用率触发的先天不足 | 无法做到"Pending 才弹" | 评估 Karpenter/CA 替代 |
| spec/annotation 版本漂移 | `non-npu-vcjob-prefer-001` 的 annotation 比 spec 多了 ascend-1980/310 检查 | 清理 annotation 或同步 |
| member1/member2 secret 策略 selector 缺少 kind 约束 | 仅匹配 `dispatch/001=true`，未限定 Secret | 加 `apiVersion: v1, kind: Secret` 到 selector |
| 001 的 ascend-1980 卡比 wlcb 多 | 标签 `npu-b` 表达不出"主 ascend 承接"角色 | 评估是否需要 `primary-for: ascend-infra` 标签 |

## 十一、回滚 SOP

```bash

TYPE=<cpp|cop|pp|hna|vcjob|workflow>

kubectl --context <ctx> get $TYPE <name> -o yaml > backup-${DATE}-${NAME}.yaml
```

### 11.2 第 1 层回滚

- **打错的标签**：`kubectl label cluster <name> <key>-`（减号删除标签）
- **cpp 改回 clusterNames**：`kubectl apply -f backup-${DATE}-<策略名>.yaml`
- **删掉的 11 条 ns 策略**：`kubectl apply -f backup-${DATE}-karmada.yaml`

### 11.3 第 2 层回滚

```bash
kubectl apply -f backup-${DATE}-non-npu-vcjob-node-affinity.yaml
```

### 11.4 第 3 层回滚

```bash
# HNA 完整 spec 恢复（ruleName 不直观，必须全量恢复）
```

### 11.5 跨层回滚顺序

若多层同时改坏：**先回滚弹性 → 再回滚调度 → 再回滚分发 → 最后回滚源头**。
下层（源头）出问题影响面最大，先回上游能快速止血。
